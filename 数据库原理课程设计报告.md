## 第1章 绪论

### 1.1 开发背景

随着经济的快速发展和人民生活水平的提高，旅游业已成为我国国民经济的重要支柱产业之一。越来越多的人选择通过旅游来放松身心、增长见识。同时，互联网的普及使得信息共享变得更加便捷，旅游攻略分享平台应运而生。

传统的旅游信息获取方式主要依赖于旅行社、旅游书籍或零散的网络信息，存在信息不全面、更新不及时、交互性差等问题。开发一个集旅游信息查询、攻略分享、用户交流于一体的旅游平台具有重要的现实意义。

### 1.2 研究意义

本系统的设计与实现具有以下意义：
1. 为旅游爱好者提供一个信息共享和交流的平台
2. 整合各城市旅游资源，提供便捷的查询服务
3. 通过用户分享的真实攻略，提高旅游体验
4. 实践数据库设计理论，将课堂知识应用于实际项目

### 1.3 研究内容

## 第2章 关键技术介绍

### 2.1 B/S架构

**简介**：B/S（Browser/Server，浏览器/服务器）架构是一种基于Web的软件架构模式。用户通过Web浏览器访问服务器端应用，无需安装专门的客户端软件。

**选型理由**：
- 客户端无需安装，使用方便
- 跨平台性好，支持Windows、Mac、Linux等系统
- 维护成本低，只需更新服务器端即可
- 便于部署到云端（如Render、Railway等平台）

### 2.2 Node.js + Express后端框架

**Node.js简介**：Node.js是一个基于Chrome V8引擎的JavaScript运行环境，可用于开发高性能的网络应用。

**Express框架**：Express是基于Node.js的轻量级Web应用框架，提供了简洁的路由、中间件等功能。

**在项目中的应用**：
- 处理HTTP请求与响应
- 实现RESTful API接口
- 提供静态文件服务
- 用户认证与授权

### 2.3 前端技术

**HTML5/CSS3/JavaScript**：采用原生前端技术，无需引入复杂框架，学习成本低，性能优秀。

**设计风格**：采用中国风UI设计，使用传统配色（朱砂红、金色），营造文化氛围。

### 2.4 数据存储方案

**JSON文件存储**：本系统初期采用JSON文件作为数据存储方案，便于开发和测试。后期可平滑迁移到MySQL等关系型数据库。

## 第4章 数据库概念结构设计 ★★

### 4.1 实体与属性分析

从需求中抽象出以下实体：

**1. 用户（users）**
- `id`：用户ID（主键）
- `username`：用户名（唯一）
- `password`：密码（加密存储）
- `nickname`：昵称
- `avatar`：头像
- `bio`：个人简介
- `level`：用户等级
- `experience`：经验值
- `created_at`：注册时间
- `last_login`：最后登录时间
- `is_online`：是否在线

**2. 管理员（admins）**
- `id`：管理员ID（主键）
- `username`：用户名（唯一）
- `password`：密码（加密存储）
- `nickname`：昵称
- `avatar`：头像
- `level`：管理员等级
- `created_at`：创建时间
- `last_login`：最后登录时间

**3. 省份（provinces）**
- `id`：省份ID（主键）
- `name`：省份名称
- `region`：所属区域（华北、华东、华南、西南、西北、东北、港澳台）

**4. 城市（cities）**
- `id`：城市ID（主键）
- `name`：城市名称
- `province_id`：所属省份ID（外键）
- `province_name`：省份名称
- `highlights`：景点亮点（逗号分隔）
- `description`：城市描述
- `image`：城市图片URL
- `best_season`：最佳旅游季节
- `view_count`：浏览次数

**5. 攻略（posts）**
- `id`：攻略ID（主键）
- `user_id`：作者用户ID（外键）
- `title`：攻略标题
- `content`：攻略内容
- `images`：图片URL（逗号分隔）
- `city_id`：关联城市ID（外键）
- `likes_count`：点赞数
- `comments_count`：评论数
- `created_at`：发布时间

**6. 评论（comments）**
- `id`：评论ID（主键）
- `post_id`：所属攻略ID（外键）
- `user_id`：评论者用户ID（外键）
- `content`：评论内容
- `created_at`：评论时间

**7. 点赞（likes）**
- `id`：点赞ID（主键）
- `post_id`：被点赞的攻略ID（外键）
- `user_id`：点赞者用户ID（外键）
- `created_at`：点赞时间

## 第5章 数据库逻辑与物理设计 ★★★

### 5.1 逻辑结构设计

#### E-R图向关系模式转换

将E-R图转换为以下关系模式（主键用下划线标注）：

**1. 用户表**
```
users(_id_, username, password, nickname, avatar, bio, level, experience, created_at, last_login, is_online)
```

**2. 管理员表**
```
admins(_id_, username, password, nickname, avatar, level, created_at, last_login)
```

**3. 省份表**
```
provinces(_id_, name, region)
```

**4. 城市表**
```
cities(_id_, name, province_id, province_name, highlights, description, image, best_season, view_count)
外键：province_id → provinces(id)
```

**5. 攻略表**
```
posts(_id_, user_id, title, content, images, city_id, likes_count, comments_count, created_at)
外键：user_id → users(id)
外键：city_id → cities(id)
```

**6. 评论表**
```
comments(_id_, post_id, user_id, content, created_at)
外键：post_id → posts(id)
外键：user_id → users(id)
```

**7. 点赞表**
```
likes(_id_, post_id, user_id, created_at)
外键：post_id → posts(id)
外键：user_id → users(id)
```

#### 规范化分析

**用户表（users）- 3NF验证**
- 主键：id
- 所有非主键属性完全依赖于主键
- 不存在传递依赖
- **结论**：满足3NF

**城市表（cities）- 3NF验证**
- 主键：id
- 所有非主键属性完全依赖于主键
- province_name依赖于province_id，但province_id是外键，属于合理冗余（为了查询性能）
- **结论**：满足3NF

**攻略表（posts）- 3NF验证**
- 主键：id
- 所有非主键属性完全依赖于主键
- 不存在传递依赖
- **结论**：满足3NF

其他表同理，均满足3NF要求。

### 5.5 触发器设计

**触发器1：更新攻略点赞数（trg_update_likes_count）**

- **触发时机**：在likes表INSERT/DELETE后
- **核心逻辑**：自动更新posts表的likes_count字段
- **SQL**：
```sql
-- 插入点赞后
CREATE TRIGGER trg_after_like_insert
AFTER INSERT ON likes
FOR EACH ROW
UPDATE posts SET likes_count = likes_count + 1 WHERE id = NEW.post_id;

-- 删除点赞后
CREATE TRIGGER trg_after_like_delete
AFTER DELETE ON likes
FOR EACH ROW
UPDATE posts SET likes_count = likes_count - 1 WHERE id = OLD.post_id;
```

#### 6.2.2 用户登录功能的实现

**功能描述**：用户输入用户名和密码登录，系统验证密码，生成JWT Token。

**核心代码**：

```javascript
app.post('/api/auth/login', async (req, res) => {
    const { username, password } = req.body;
    
    // 查询用户
    const user = findByField(db.users, 'username', username);
    if (!user) {
        return res.status(401).json({ error: '用户名或密码错误' });
    }
    
    // 验证密码
    const isMatch = await bcrypt.compare(password, user.password);
    if (!isMatch) {
        return res.status(401).json({ error: '用户名或密码错误' });
    }
    
    // 更新登录状态
    user.last_login = new Date().toISOString();
    user.is_online = 1;
    saveDb();
    
    // 生成JWT Token
    const token = jwt.sign({ userId: user.id }, JWT_SECRET, { expiresIn: '30d' });
    
    res.json({ success: true, token, user });
});
```
